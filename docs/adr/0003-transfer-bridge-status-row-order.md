# ADR-0003: Transfer 桥接状态以 transactions 行顺序为权威

- 状态：Accepted
- 日期：2026-05-09

## 背景

Polymarket bridge `status/{address}` 返回的 `transactions` 用于 Transfer Crypto 底部状态（等待转账、处理中、已入账等）。实现过程中曾用各行的 `createdTimeMs` 取最大值来推断「当前最新状态」，但在 **`DEPOSIT_DETECTED` 阶段** 观察到：

- 响应里 **可能没有 `txHash`**；
- **可能没有 `createdTimeMs`**。

在此条件下，无法用时间戳或哈希在多条记录之间建立稳定的全序；若仍用 `createdTimeMs` 数值比较，会把无时间戳的行当成 0，被其它行更大时间戳 **错误覆盖**，导致跳过「处理中」或与官网表现不一致。

## 决策

1. **信任上游 `transactions` 数组顺序**：约定为 **新在前、旧在后**（与当前观测及官网行为一致）。
2. 在应用 **Transfer UI Session Baseline** 与会址时间窗口过滤之后，**取过滤结果数组的第一条** 的状态作为当前展示阶段，**不再**用 `createdTimeMs` 取最大值代替「最新」。
3. 若上游顺序契约在未来变更，须同步修订展示逻辑或引入可关联的稳定键（例如每笔流水的唯一 id），并新增 ADR。

## 影响

- Transfer 状态栏与轮询逻辑与「首行即当前阶段」一致；`DEPOSIT_DETECTED` 无字段时仍可依赖顺序显示「处理中」。
- 若某次响应中行顺序与约定不符，展示可能短暂不一致；风险归因为上游契约，而非前端单行启发式。

## 权衡

- 优点：与可得字段一致，避免无时间戳检测态被误合并；与官网「只看最新一条是否为 `DEPOSIT_DETECTED`」同构。
- 代价：**强依赖** `transactions` 排序契约；排序若不可靠则需后端裁剪或上游修复。

## 后续动作

- 若在测试环境捕获到顺序异常样本，记录请求时间与原始 JSON，评估是否需要在代理层排序或过滤。
